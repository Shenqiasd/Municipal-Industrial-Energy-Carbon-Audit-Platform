中等问题 (Medium)
#2. data-h2.sql BCrypt hash 使用 $2b$ 前缀，Java BCrypt 库可能不兼容

'$2b$10$0T0mZTyDLfErwhn/qSjIG.MlhJOt9BdDpKQ58iXLRMNlwqI2pmbpK'
Spring Security 的 BCryptPasswordEncoder 使用 $2a$ 前缀。$2b$ 是某些非 Java 工具（如 Node.js bcrypt）生成的。虽然部分版本的 Spring Security 可以兼容 $2b$，但存在兼容性风险。

建议: 使用 Java 端 BCryptPasswordEncoder 生成 hash 并替换，或确认当前使用的 Spring Security 版本已支持 $2b$。

#3. application.yml 默认激活 dev profile

spring:
  profiles:
    active: dev
主配置文件默认 active: dev，意味着如果忘记设置 --spring.profiles.active=prod，生产部署也会使用 H2 内存数据库。PR 描述中说"生产环境不受影响"，但这需要运维人员主动指定 prod profile。

建议: 默认不设置 active，或在 docker-compose.yml / Dockerfile.backend 中显式设置 SPRING_PROFILES_ACTIVE=prod。

#4. H2 作为无条件依赖打入生产 JAR

<!-- audit-web/pom.xml -->
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <scope>runtime</scope>
</dependency>
注释说明了这是为开发便利，但 scope=runtime 意味着 H2 驱动会被打入生产 fat JAR（约 2.5MB）。结合 #3，如果生产环境未指定 prod profile，将直接使用 H2 内存库。

建议: 如果不想引入 Maven Profile 复杂度，至少确保 docker-compose.yml 或 Dockerfile.backend 强制指定 SPRING_PROFILES_ACTIVE=prod。

#5. vite.config.ts 设置 allowedHosts: true 禁用了 Host 校验

server: {
    host: '0.0.0.0',
    port: 5000,
    allowedHosts: true,  // <-- 禁用所有 Host 校验
allowedHosts: true 等价于允许任何 Host 头的请求，这在 Replit 环境中是必要的，但如果此配置被带到其他开发/测试环境中，存在 DNS rebinding 攻击风险。

建议: 加注释说明这是 Replit 专用配置，或改为 allowedHosts: ['*.replit.dev']。

#6. Mapper XML deleteById 缺少审计追踪 (5个 Mapper 受影响)
PR #1 的审核中已指出 SysUserMapper.xml 的 deleteById 应记录 update_by，且 PR #1 已修复此问题。但 PR #2 新增的 5 个 Mapper XML 的 deleteById 全部没有设置 update_by：


<!-- EntEnterpriseMapper.xml, BsEnergyMapper.xml, BsProductMapper.xml, BsUnitMapper.xml, SysRoleMapper.xml -->
<update id="deleteById">
    UPDATE xxx SET deleted = 1, update_time = NOW() WHERE id = #{id}
</update>
对比已修复的 SysUserMapper.xml:


<update id="deleteById">
    UPDATE sys_user SET deleted = 1, update_by = #{updateBy}, update_time = NOW() WHERE id = #{id}
</update>
影响: 软删除操作无法追溯是谁执行的。PR #1 修复了这个问题后，PR #2 又重复引入了同样的问题。

建议: 所有 deleteById 增加 @Param("updateBy") String updateBy 参数和 update_by = #{updateBy} 字段。同时需要修改对应的 Java Mapper 接口签名。

#7. TplTemplateMapper.xml 的 deleteById 同样缺少审计追踪
与 #6 相同，但此处还需特别注意：TplTemplateMapper.xml 缺少 update_by，且对应的 TplTemplateMapper.java 接口签名为 deleteById(@Param("id") Long id)，需同步修改。

非阻塞建议 (Non-blocking)
A. package.json 仅做依赖排序
此 PR 对 package.json 的变更仅是将依赖按字母排序。功能正确，但这类纯格式化变更引入了大量无意义的 diff（以及 3139 行的 package-lock.json）。建议后续将格式化变更单独提交或使用 fixpack 工具统一处理。

B. .replit 和 replit.md 是 Replit 平台特定文件
这些文件仅在 Replit 平台上有意义。如果团队不使用 Replit，建议加入 .gitignore。如果使用，则当前配置合理。

C. auto-imports.d.ts 和 components.d.ts 是自动生成文件
这些文件由 unplugin-auto-import 和 unplugin-vue-components 自动生成。通常应加入 .gitignore 以避免 merge conflict。如果团队有意提交这些文件（为了 IDE 类型提示），则可接受。

D. schema-h2.sql 缺少 sys_user_role 的 create_by/create_time 列
H2 中的 sys_user_role 只有 id, user_id, role_id，但生产 MySQL 还有 create_by, create_time, update_by, update_time, deleted。如果后续有 Mapper XML 操作此表并引用这些列，H2 dev 环境会报错。
