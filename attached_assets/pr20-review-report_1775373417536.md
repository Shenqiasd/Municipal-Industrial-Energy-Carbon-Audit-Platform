#### C1. EnterpriseController lock/unlock/expire 端点缺少管理员权限校验

**严重等级**: Critical  
**文件**: `audit-web/src/main/java/com/energy/audit/web/controller/enterprise/EnterpriseController.java:80-108`

**问题描述**: `lock()`、`unlock()` 和 `updateExpireDate()` 三个端点没有任何角色权限校验。任何已认证用户（包括企业用户 userType=3、审核员 userType=2）均可锁定/解锁任意企业或修改过期日期。同一 PR 中 `TemplateController` 已正确使用 `requireAdmin()` 守卫模式，但 `EnterpriseController` 未应用。

此外，`create()`、`update()`、`delete()` 同样缺少 `requireAdmin()` 校验。

```java
// 当前代码 — 无权限检查
@PutMapping("/{id}/lock")
public R<Void> lock(@PathVariable Long id) {
    enterpriseService.lock(id);
    return R.ok();
}
```

**修复建议**: 参照 `TemplateController` 添加 `requireAdmin()` 方法，在所有写操作端点首行调用。

> **注: 与 Devin Review Comment #1 一致 — 确认有效**

#### C2. TemplateVersionServiceImpl 全部方法为空壳（stub），导致模板创建→发布流程完全不可用

**严重等级**: Critical  
**文件**: `audit-service/src/main/java/com/energy/audit/service/template/impl/TemplateVersionServiceImpl.java`

**问题描述**: 整个 `TemplateVersionServiceImpl` 是空壳实现：
- `getById()` 返回 `null`
- `listByTemplateId()` / `listVersionsMeta()` 返回空列表
- `create()` / `saveJson()` / `publish()` / `update()` / `delete()` 为空方法
- `createDraftVersion()` 创建内存对象但**从不持久化到数据库**

这导致：
1. 管理员创建模板时，`TemplateServiceImpl.create()` 调用 `versionService.createDraftVersion()` → 版本不入库
2. 发布模板时，查询版本列表为空 → 永远抛出 "没有可发布的草稿版本"
3. `syncTagsFromJson` 调用 `versionService.getById()` 返回 `null` → **NPE (500 错误)**

**涉及调用链**:
```
TemplateController.syncTagsFromJson()
  → versionService.getById(versionId)   // returns null
  → version.getTemplateJson()           // NPE!
```

> **注: 与 Devin Review Comment #3 和 #4 一致 — 确认有效**

#### C5. ChartDataController 缺少角色权限校验 — 任何已认证用户均可调用

**严重等级**: Critical  
**文件**: `audit-web/src/main/java/com/energy/audit/web/controller/data/ChartDataController.java`

**问题描述**: ChartDataController 的 5 个端点使用 `SecurityUtils.getRequiredCurrentEnterpriseId()` 获取企业隔离 ID，但**未校验用户角色**。管理员（userType=1）和审核员（userType=2）的 `enterpriseId` 为 `null`，调用时会抛出 403 异常（"当前用户不属于任何企业"）。但更严重的是：

**建议**: 添加显式的 `requireEnterprise()` 角色守卫，明确此端点仅限企业用户访问。

#### M3. H2 schema 与 MySQL schema 四层对齐偏差 — de_energy_balance 缺少 energy_name/energy_category 列

**严重等级**: Medium  
**文件**: `sql/00-schema.sql:819-838` vs `audit-web/src/main/resources/schema-h2.sql:224-244`

**问题描述**: MySQL schema 中 `de_energy_balance` 表的列与 H2 schema 存在差异：

| 列名 | MySQL (00-schema.sql) | H2 (schema-h2.sql) |
|------|----------------------|---------------------|
| `energy_name` | **缺失** | 存在 VARCHAR(128) |
| `energy_category` | **缺失** | 存在 VARCHAR(64) |
| `standard_coal_equiv` | **缺失** | 存在 DECIMAL(18,4) |

MySQL 版本仅有 `energy_id` (外键引用 bs_energy)，而 H2 版本额外冗余了 `energy_name`、`energy_category`、`standard_coal_equiv`。ChartDataController 的 SQL 查询直接使用 `energy_name`、`energy_category`、`standard_coal_equiv` 列名：

```sql
SELECT energy_name AS name, standard_coal_equiv AS value, energy_category AS category
FROM de_energy_balance WHERE enterprise_id = ? AND audit_year = ? AND deleted = 0
```

**这意味着在 MySQL 生产环境中这个查询会直接报错（列不存在）。** H2 开发环境因为有冗余列所以能工作，但生产环境（MySQL）会崩溃。

## 三、SQL 注入安全审计

### ChartDataController — 所有 5 个端点 SQL 注入审查结果

| 端点 | 方法 | 参数化查询 | 结论 |
|------|------|-----------|------|
| `/energy-structure` | `jdbcTemplate.queryForList(sql, enterpriseId, auditYear)` | 使用 `?` 占位符 | **安全** |
| `/energy-trend` | `jdbcTemplate.queryForList(sql, enterpriseId, auditYear)` | 使用 `?` 占位符 | **安全** |
| `/product-consumption` | `jdbcTemplate.queryForList(sql, enterpriseId, auditYear)` | 使用 `?` 占位符 | **安全** |
| `/ghg-emission` | `jdbcTemplate.queryForList(sql, enterpriseId, auditYear)` | 使用 `?` 占位符 | **安全** |
| `/summary` | `jdbcTemplate.queryForMap(sql, enterpriseId, auditYear, auditYear)` | 使用 `?` 占位符 | **安全** |

**结论**: 所有 5 个 JdbcTemplate 查询均使用参数化查询（`?` 占位符 + varargs 传参），**不存在 SQL 注入风险**。`enterpriseId` 从 JWT ThreadLocal 获取，`auditYear` 为 `Integer` 类型，均非用户可控字符串拼接。

### MyBatis Mapper XML 注入审查

所有 Mapper XML 使用 `#{param}` 预编译参数（非 `${param}` 拼接），**安全**。
